## Decisions

- 21:05 - Treat active run liveness as a two-signal check: operator surfaces should require both a reachable supervising PID and a fresh persisted heartbeat so stale `running` records do not become active again after PID reuse.
- 21:05 - When a detached worker supervisor cannot spawn, persist a terminal error in the prepared run directory instead of crashing and leaving `run.md` stuck in `running`.
- 19:04 - Use provider CLI MCP inspection as the source of truth for required-MCP preflight instead of reading provider settings files directly, because the CLI can confirm real connectivity rather than just configuration presence.
- 19:12 - Treat Codex and Gemini MCP readiness differently based on what their CLIs actually expose today: Gemini can fail on disconnected MCPs directly, while Codex preflight currently treats `enabled` in `codex mcp list` as the strongest available ready signal.
- 19:45 - Prefer structured provider MCP output when available; for Codex, `codex mcp list --json` is a better contract than scraping the human-readable table.
- 18:07 - Keep the public CLI grouped under `agent`, `skill`, `run`, and `sesh`; do not keep the older flat public command names, and use `sesh` as the operator-facing namespace for both live and completed sessions.
- 17:00 - Keep description truncation presentation-only: shorten long descriptions in human-readable tables for `list` and `skills`, but leave JSON payloads and underlying metadata untouched.
- 16:57 - Initial direction was to keep skill discovery flat with a top-level `aiman skills` command; superseded at 18:07 by the grouped `aiman skill list` public command tree.
- 16:52 - Keep skill execution provider-native: `aiman` should validate declared skills from `.agents/skills/` and `~/.agents/skills/`, prefer project scope on collisions, and record the resolved files in launch metadata rather than composing `SKILL.md` contents into provider prompts itself.
- 17:03 - Keep `aiman top` as a three-view dashboard: default to `active`, add explicit `historic`, preserve `all`, and define historic as any non-active run including stale pid-backed `running` records.

- 16:12 - Treat `aiman` as a provider-neutral specialist-run recorder: the core product is one authored specialist run with a trustworthy persisted record, not agent orchestration or workflow control.
- 16:22 - Keep detached execution as the same run contract with a different launch path; hidden workers should rebuild from frozen `launch` evidence plus `prompt.md`, not live agent files or fresh provider-specific derivation.

- 13:36 - Treat agent YAML as the source of truth for execution permissions: each agent must declare `read-only` or `workspace-write`, and callers should not be able to widen or narrow that permission silently at launch time.
- 13:28 - Treat provider rights as first-class operator data: callers should not have to infer read-only vs write-enabled behavior from raw adapter flags or mode names alone.
- 13:01 - Prefer explicit `--instructions` over waiting on stdin in `aiman create`; parent agents and wrappers should not have to close an otherwise unused stdin pipe just to avoid a hang.

- 12:10 - Prefer the simpler pre-trace runtime contract for now; do not add agent-authored trace requirements or trace-specific run failure modes until there is a lower-friction design.
- 10:20 - Keep `top` text-first but make it feel like a dashboard: explicit panes, stronger hierarchy, and concise per-run summaries are worth more than squeezing in more raw fields.
- 10:14 - Prefer a simple raw-terminal dashboard over `neo-blessed` for `aiman top`; stability and predictable TTY behavior matter more here than a richer widget layout.
- 10:06 - Prefer a plain `blessed.list` over `neo-blessed` `listtable` for the `top` run picker because `listtable` crashes during initial shrink-layout rendering in a real TTY in this environment.
- 22:30 - Treat detached launches as snapshotted work: once `run --detach` returns, the worker should rely on persisted launch metadata and prompt content rather than re-reading mutable agent files by name.
- 22:52 - Keep successful empty foreground runs silent instead of inventing a fallback summary; the human path should only print the final answer when one exists and otherwise avoid contradicting the persisted success status.
- 00:11 - To remove the runtime footer without dropping task input entirely, add explicit placeholder substitution owned by the agent body instead of continuing to append hidden runtime text.
- 00:18 - Keep placeholder rendering explicit and minimal: support `{{task}}` plus a small set of optional runtime placeholders, but only substitute values the agent body explicitly requests.
- 00:25 - Do not break browsing/listing of older agent files; only `run` should fail on missing `{{task}}`, with a clear validation message telling the author what to add.
