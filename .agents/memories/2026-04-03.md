## Decisions

- 12:42 - The default app should stop surfacing the old skills-selection path in its main flow for now; v1 task creation is intentionally a single-agent-plus-task launcher, while agent inspection stays read-only on its own page.
- 12:49 - The shared header should render the hotkeys unchanged in the top band, then a second band with `PAGE ...` on the left and global run-health counters on the right so users can always orient themselves without sacrificing compactness.
- 12:53 - The runs page should list the global indexed run store, show project names alongside each run, and sort active runs ahead of historic rows so the workbench stays useful across repositories.
- 12:32 - The default app should become a four-page top-level workbench (`home`, `agents`, `tasks`, `runs`) with nested read-only agent details and run inspection, rather than keeping the current mixed agent/task flow and separate history/live concepts.
- 12:33 - The shared header should keep the existing hotkey row, add a compact ASCII brand mark, show the current page underneath, and expose global run health with a slow-blinking green ASCII running indicator plus a solid red failed indicator.
- 12:33 - The new task flow should stay intentionally simple for v1: choose agent, write task, and start the run, without keeping the current skill-selection UI in the main path.
- 01:56 - Main-app `Esc` behavior should be parent-oriented, not recency-oriented: inspector siblings should collapse back to the underlying details page, and details should collapse to the nearest base page such as runs or agents.
- 01:38 - TUI height should come from the actual terminal rows minus real header/footer usage, not from a stale fixed subtraction, so the app and `sesh top` fill tall terminals cleanly.
- 01:23 - `reasoningEffort` is part of the required authored profile contract, but its valid values are provider-specific: Codex allows `none|low|medium|high`, while Gemini currently requires `none`.
- 01:23 - Provider-specific `reasoningEffort` validation must happen before `profile create` writes the file, so invalid provider/value combinations fail cleanly without leaving a bad profile on disk.
- 11:41 - The public `aiman` profile contract should be strict and forward-only: authored profiles must use `mode`, legacy frontmatter fields should fail validation, and the public `profile migrate` path should be removed rather than documented.
- 11:55 - The remaining compatibility aliases may persist in some internal run metadata and hidden compatibility modules for now, but the public profile loader, docs, and CLI surface must reject legacy authored frontmatter instead of tolerating it.
- 11:03 - The new authoring skill and the refreshed docs should describe the live `aiman profile` contract and the auto-attached `AGENTS.md#Aiman Runtime Context`, not the older `agent`/`contextFiles` authoring model.
- 11:16 - The authoring helper for future Codex sessions should live under tracked repo skills at `.agents/skills/`, while the product docs should still describe `aiman`'s separate local runtime skills under `.aiman/skills/`.
- 09:07 - The shared header should be reduced to branded ASCII plus wrapped hotkeys only; project-level summary data belongs on the home screen, not in persistent header chrome.
- 14:05 - The interactive CLI is now Bun-native and OpenTUI React-based; keep `aiman` as the only human TTY entrypoint and remove the separate `sesh top` dashboard instead of preserving both surfaces.
- 14:22 - The unified workbench should be explicitly two-workspace and keyboard-first: `launch` for profile/task entry and `runs` for active/historic inspection, with stop/reuse actions kept in the same TTY surface.
