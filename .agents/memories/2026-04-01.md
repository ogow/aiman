## Decisions

- 19:39 - A declared required MCP being enabled in provider config is not enough to guarantee the authored Codex agent will actually use it. The inspected `chrome_web_explorer` failure came from the model choosing shell/repo fallback without first attempting `chrome-devtools`, even though the project config exposed `chrome-devtools` and a direct Codex run could call it.
- 15:56 - The right boundary for Codex-backed `aiman` runs is to keep project-scoped provider config available for MCP registration, but explicitly blank the project-doc and developer-instruction inputs that would otherwise pollute authored `aiman` agent prompts.
- 16:10 - The right boundary for Gemini-backed `aiman` runs is similar, but Gemini needs a different mechanism: keep project `.gemini/settings.json` active for MCP registration, while overriding `context.fileName` through a per-run system settings file so workspace context files like `AGENTS.md` or `GEMINI.md` do not enter the authored `aiman` prompt.
- 20:24 - Gemini's current CLI surface is strong enough to make agent `skills:` deterministic: the generated per-run system settings file can keep project MCP config and context isolation intact while setting `skills.enabled`/`skills.disabled` so only the agent-declared Gemini skills remain available during that run.
- 19:28 - Windows Codex runs should explicitly disable login-shell semantics and user shell profile loading in `src/lib/providers/codex.ts` so non-interactive `aiman` runs do not inherit PowerShell profile failures from the host machine. The remaining constrained-language stderr under `pwsh -NoProfile` is a separate Codex/PowerShell behavior and should not block removing the profile-loading failure now.
- 17:42 - In `sesh top`, detail view should own its own scroll state; list navigation keys must scroll the selected run's content while detail view is open instead of moving the run selection underneath it. The final-answer section should also do lightweight line-oriented Markdown rendering rather than dumping raw Markdown verbatim.
- 17:05 - Home-level user scope directories must never promote `$HOME` into project scope; only project-local `.aiman` / `.agents` directories below home, or non-home markers such as `.git` and `AGENTS.md`, may define the project root.
- 17:12 - When a provider launch goes through Windows `cmd.exe /d /s /c` for a `.cmd` or `.bat` shim, stop and timeout handling must terminate that Windows process tree instead of only killing the command-processor wrapper.
- 12:40 - For v1, move only runs/session state into `~/.aiman`, keep project/user agent and skill resolution unchanged, use SQLite as an index rather than a blob store, and do not auto-import legacy project-local runs.
- 14:20 - Keep run contents file-backed and human-inspectable (`run.md`, `prompt.md`, logs, artifacts), but treat the SQLite database as the authoritative global lookup path for listing and resolving runs from any cwd.
- 12:18 - Treat the nearest ancestor containing project markers as the effective project root; `aiman` should not require callers to `cd` to the exact repo root just to inspect active sessions.
- 11:20 - Stop using `shell: true` for Windows `.cmd` and `.bat` provider launches; instead, translate those launches into an explicit escaped `cmd.exe /d /s /c` command line so prompt arguments survive Windows metacharacter parsing.
- 11:23 - Keep the Windows verification honest by making the CLI fixtures and runnable-provider test helpers genuinely cross-platform instead of relying on missing static fixtures or `/bin/sh`.
- 10:58 - On Windows, search `PATHEXT` candidates before the bare command name so npm-style `.cmd` wrappers win over adjacent extensionless shim files that are present on disk but not directly runnable.
- 10:15 - Fix provider launch in one shared place instead of per adapter: executable resolution should understand Windows `PATHEXT` shims, and both MCP preflight plus actual provider execution should go through the same resolved-launch helper.
- 10:34 - Keep the persisted launch snapshot provider-facing and stable by recording logical commands like `codex`/`gemini`, while resolving the concrete executable path only at execution time.
- 10:43 - Treat `HOME` as an explicit user-scope override on every platform so tests, sandboxes, and real Windows setups can redirect `~/.aiman` and `~/.agents` consistently without patching call sites.
- 09:29 - Treat missing `SKILL.md` inside a discovered skill directory as a stale or incomplete install that `aiman skill list` should skip, while preserving hard failures for other file-read or parsing errors.
- 09:35 - Keep the missing-file tolerance scoped to listing only; installation and other code paths should still use the strict skill reader so real install problems remain visible.
- 09:44 - For git skill installs, follow the source repo's default branch rather than hard-coding `main`; the installer should work against ordinary git remotes without requiring branch-name conventions.
- 09:49 - Keep git skill installs repo-agnostic for now: clone the default branch through ordinary git behavior instead of adding host-specific archive-download logic.
